Systems Check

AI replaces the typing, not the understanding. These are the questions that keep you owning the thinking when a machine writes the code.

Seven diagnostic lenses for figuring out what the AI actually did to your project — not just whether it runs.

The rule for all of these: you should be able to answer without running the code. The program is the model in your head; the source is only its shadow. Any “I don’t know” isn’t a failure — it’s the next thing to learn before you ship.

The seven lenses L1 – L7
L1 · STATE & OWNERSHIP

Where does the truth live?

Two owners for one fact is a bug that just hasn’t fired yet.

  • For each piece of data touched — who owns it? DB, cache, client, or external service?
  • Is there a single source of truth, or several copies drifting apart?
  • Where did the AI silently duplicate state “for convenience”? Find it.
L2 · FEEDBACK & OBSERVABILITY

How do I know it works?

Code that can’t be seen failing is only pretending to work.

  • What signal tells you the system is healthy versus broken?
  • Where do errors go — handled, logged, or silently swallowed?
  • Can you actually watch it fail, or does it collapse without warning?
L3 · BLAST RADIUS

What breaks if I touch this?

The AI optimizes the local change. You own the ripple.

  • If this component changes, what else must change with it?
  • What depends on it that the diff never touches?
  • “If I delete or modify X, what fails and how badly?”
answer “I don’t know” → close it before shipping
L4 · THE DRAW-IT TEST

Can I draw it?

What you can’t draw is usually where it goes wrong.

  • On paper: a box per component, an arrow per data flow.
  • Mark where state lives and where failures can appear.
  • If you can’t sketch it, you don’t understand it yet.
L5 · SPEC FIT

What & why, before how?

Right code for a slightly wrong problem is still wrong.

  • What is this supposed to do, and why?
  • What are the constraints, success criteria, and failure modes?
  • Did the AI solve the problem you meant — or one next to it?
L6 · INTERROGATE THE CHOICES

Deterministic or not?

A compiler you trust without reading. AI you don’t.

  • “Walk me through each step and the alternatives you considered.”
  • Why this approach over the others — what breaks if we invert it?
  • Did it slip in a security hole, race condition, or wrong business rule?
L7 · JAGGED-FRONTIER ZONES

Where is AI confidently wrong?

Brilliant in one spot, quietly broken in the next — same session.

— · THE COMPILER TEST

Why AI isn’t “just the next abstraction”

Abstraction only helps when the layer below is trustworthy.

  • A compiler is deterministic: same input, same correct output — trust without understanding.
  • AI is probabilistic: different output each time, no guarantee.
  • So it’s a collaborator, not a compiler — trust it only as far as you understand it.
Keep the muscle ongoing
Weekly · by hand

Rewrite one AI-generated chunk manually. The slow, deliberate writing is what builds the mental model — the part the AI can’t hand you.

Periodic · audit

Pick any component you touched recently and ask “if I break this, what happens?” Every “I don’t know” is a study assignment.

Pre-ship gut check 30 seconds
Before you merge

Five questions, no code running

  1. Where does state live, and who owns it?
  2. How would I know if this broke?
  3. What is the blast radius of this change?
  4. Could I draw this system on paper right now?
  5. Do I understand why the AI did it this way?
